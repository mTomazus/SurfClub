# Forecast Notification (Telegram) — Final Decisions

Page: `modules/test/views/port.php` (route `test/port`). Controller `modules/test/controllers/Test.php`.
Existing: `surf_forecast` (4-day, per-day numeric fields + AI word), `surf_rating` (current).
Forecast day fields: date, label, wave_min, wave_max, wave_period, wind_avg, wind_max, wind_dir(compass), temp_max, spark.
Host: Hostinger shared (surfclub.lt). Cron available. cURL used throughout (good for Bot API).

## GOAL
Let visitors of test/port subscribe to a "tomorrow looks good" alert, personalised to their level, delivered via Telegram.

## CHANNEL
- Telegram bot. Chosen over web push (iOS needs PWA install) and email. Universal reach incl. iPhone, free, instant; cadence is non-urgent so that's fine.
- No service worker / VAPID / Composer lib. Just cURL to Bot API + a webhook endpoint.
- All messages in Lithuanian.

## TRIGGER
- Timing: UPCOMING/forecast (not live). Source = surf_forecast.
- Window: TOMORROW ONLY (the day = today+1).
- Match: PURE NUMERIC, AND across set stats. AI word = on-page decoration only.
- Profile stats: wave height range, wind speed max, wind direction (favourable set), wave period min.
- Favourable wind dirs: PV(SW), P(S), V(W), ŠV(NW) — FLAT boolean for v1.
- Domain note (v2, NOT v1): quality is a gradient peaking at S — closer to S = good→great. Ranked by closeness to S(180°): P(S) best → PV(SW) → V(W) → ŠV(NW). v2 could use this to label "puikios" vs "geros" in the message and/or gate Advanced to near-S days. Decided: IGNORE for v1.

## PROFILES (hardcoded presets, tweak via code)
- Beginner: wave 0.4–1.0 m · wind ≤ 6 m/s · period any · dir SW/S/W/NW
- Advanced: wave ≥ 1.2 m · wind ≤ 12 m/s · period ≥ 4 s · dir SW/S/W/NW
- Custom: user sets all four in the modal.

## STORAGE / IDENTITY
- Anonymous. New table (e.g. surf_alerts): id, chat_id(null until linked), profile_type, custom wave_min/wave_max/wind_max/period_min, dirs, link_token, status(pending/active/stopped), last_notified_date, created_at.
- Dedup naturally trivial (one evening run, one tomorrow). last_notified_date guards re-sends within a run / re-runs.

## ONBOARDING FLOW
1. Bell on test/port → modal (Trongate MX). Pick beginner/advanced/custom; custom reveals 4 inputs.
2. "Connect Telegram" → POST profile → server creates pending sub + one-time token (TTL ~30 min, single use).
3. Deep link t.me/<bot>?start=<token>. User taps Start.
4. Webhook receives /start <token> → match token → attach chat_id, mark active (dedup by chat_id: update if exists) → bot confirms in LT.

## MESSAGE (evening alert)
- Core stats only: tomorrow's wave range + wind speed & direction. + /stop hint.
- Example: "🏄 Rytoj Melnragėje geros sąlygos! Bangos 1.0–1.4 m, vėjas 8 m/s iš PV. Parašyk /stop, kad atsisakytum."
- No link, no period/temp/AI advice.

## UNSUBSCRIBE / MANAGE
- Bot /stop → status=stopped (skip in cron).
- Change profile / re-subscribe → reopen web modal, reconnect (dedup by chat_id updates the row).
- No in-bot profile editing for v1.

## SCHEDULER
- Hostinger cron, EVENING ONLY (~18:00 Europe/Vilnius).
- Job: refresh forecast → for each active sub, check if tomorrow's numbers satisfy profile → if yes & not already notified for that date → Bot API sendMessage → set last_notified_date.
- Forecast API fail → skip run (no false alerts).

## ADMIN
- Minimal. Presets + good wind dirs hardcoded in controller. No admin UI, no subscriber list, no test-send button for v1.

## PREREQUISITES (user action)
- Create bot via @BotFather, provide BOT TOKEN (store in config like ANTHROPIC_API_KEY).
- One-time setWebhook call to register the webhook URL.

## NOT DOING (v1)
- Web push, email, SMS. Current-conditions alerts. Multi-day window (only tomorrow). Admin tuning UI. In-bot custom entry. Member-account linking.
